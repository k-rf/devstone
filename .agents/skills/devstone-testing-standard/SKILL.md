# Devstone テスト標準

このスキルは、すべてのテスト活動がプロジェクト固有の「鉄の掟」に従い、高い信頼性と保守性を維持することを保証します。

## 1. テスト戦略 (テストピラミッド)

ピラミッド型の構成を意識し、低いレイヤーほど厚くテストを行います。

| 種類                 | 対象                   | ツール | 目的                                      |
| :------------------- | :--------------------- | :----- | :---------------------------------------- |
| **Unit Test**        | `core/domain`, `utils` | Vitest | 純粋なロジックの正当性を検証。            |
| **Integration Test** | `core/application`     | Vitest | Portをモックし、Serviceの流れを検証。     |
| **Adapter Test**     | `adapter/outbound`     | MSW    | 外部APIとの通信（変換ロジック等）を検証。 |
| **E2E Test**         | `index.ts` (Entry)     | Vitest | 全レイヤーを結合した実際の挙動を検証。    |

## 2. 核心となる規律

### 2.1. インソーステスト (絶対遵守事項)

独立したロジック（`core/domain` や `utils`）の場合、テストは必ず `if (import.meta.vitest)` ブロックを使用してソースファイル内に直接記述します。

**実装パターン:**

```typescript
// ... ロジックの実装

if (import.meta.vitest) {
  const { describe, it, expect } = import.meta.vitest;

  describe("論理的なカテゴリ (例: 正常系)", () => {
    it("ビジネス要件を反映した説明 (例: 正の整数のとき、そのまま値を返す)", () => {
      // test logic
    });
  });
}
```

**例外:**

- 膨大なモックや Layer のセットアップが必要で、ソースの可読性を著しく損なう場合は `.spec.ts` に分離します。

### 2.2. 論理的な構造化

`describe` はファイル名や関数名の繰り返しではなく、**「正常系」「異常系」「境界値テスト」** といった論理的な分類にのみ使用してください。

### 2.3. 依存関係とモック

- **Layer Overriding**: 統合テストでは Port を `TestLayer` に差し替えます。実際の I/O は絶対に行いません。
- **Internal Mocks**: `Context.Tag` を実装したシンプルなオブジェクトを使用します。
- **External Mocks (MSW)**: HTTP 通信が伴う場合は必ず MSW を使用し、ネットワーク層でモックします。

### 2.4. 予測可能性とデータ駆動

- **Predictability**: ランダムな値は避け、結果が常に一定であることを保証してください。
- **Data-Driven**: パターンが多い場合は `it.each` を活用して網羅性を高めてください。

### 2.5. 失敗系の検証 (Effect-TS)

`Effect.flip` を使用してエラーチャネルを成功チャネルに入れ替え、`Effect.runPromise` でエラーオブジェクトを直接受け取って検証してください。`Exit.match` 等の使用は冗長です。

## 3. 品質基準

- **カバレッジ 100% は最低条件です。**
- 「壊れたら必ず失敗する」意味のあるテストを記述してください。
- 複雑な `Match` パターンやエラー分岐は重点的に検証してください。
- 境界値（`NaN`, `Infinity`, 空文字、`null`/`undefined` 等）を必ず網羅してください。
