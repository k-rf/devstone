# Devstone テスト標準

このスキルは、すべてのテスト活動がプロジェクト固有の「鉄の掟」に従い、高い信頼性と保守性を維持することを保証します。

## 1. テスト戦略 (テストピラミッド)

ピラミッド型の構成を意識し、低いレイヤーほど厚くテストを行います。

| 種類                 | 対象                   | ツール | 目的                                      |
| :------------------- | :--------------------- | :----- | :---------------------------------------- |
| **Unit Test**        | `core/domain`, `utils` | Vitest | 純粋なロジックの正当性を検証。            |
| **Integration Test** | `core/application`     | Vitest | Portをモックし、Serviceの流れを検証。     |
| **Adapter Test**     | `adapter/outbound`     | MSW    | 外部APIとの通信（変換ロジック等）を検証。 |
| **E2E Test**         | `index.ts` (Entry)     | Vitest | 全レイヤーを結合した実際の挙動を検証。    |

## 2. 核心となる規律

### 2.1. インソーステスト (絶対遵守事項)

モックや `faker` を必要としない、純粋なロジックやスキーマ定義のテストは、必ず `if (import.meta.vitest)` ブロックを使用してソースファイル内に直接記述します。

**例外:**

- 膨大なモックや Layer のセットアップが必要、あるいは `faker` によるテストデータの自動生成が必要で、
  ソースの可読性を著しく損なう場合は `.spec.ts` に分離します。

### 2.2. 論理的な構造化

- `describe` はファイル名や関数名の繰り返しではなく、テスト対象の論理的な分類（状況や条件によるグループ化）にのみ使用してください。
  - **ただし、単一のコンテキストしか存在しない場合は、`describe` によるグループ化を省略し、テストケース（`it`）を直接記述します。**
- テストケース名（`it` の説明）は、ビジネス要件を反映した分かりやすい**日本語**で記述してください。
- **型テスト (Type Testing)**: `.spec-d.ts` ファイルにおける型テストでも同様に、日本語で型制約を記述し、論理的な分類が必要な場合のみ `describe` で囲んで構造化してください。

### 2.3. 依存関係とモック

- **Layer Overriding**: 統合テストでは Port を `TestLayer` に差し替えます。実際の I/O は絶対に行いません。
- **Internal Mocks**: `Context.Tag` を実装したシンプルなオブジェクトを使用します。
- **External Mocks (MSW)**: HTTP 通信が伴う場合は必ず MSW を使用し、ネットワーク層でモックします。

### 2.4. 予測可能性とデータ駆動

- **Predictability & Faker**:
  - **インソーステスト**: ランダムな値は避け、結果が常に一定であることを保証するため、
    `faker`（ランダム値）は使用せず決定論的なテストデータを用いてください。
  - **外部テスト (`.spec.ts`)**: ロジックの大枠に関心があり、テストの本質ではない特定の値（例: ダミーのユーザー名やID）を
    詳細に記述する必要がない場合は、積極的に `faker` を使用して関心のない値を隠蔽・抽象化してください。
    これにより、テストコードが何に関心を持っているのかが明確になります。
- **Data-Driven**: パターンが多い場合は `it.each` を活用して網羅性を高めてください。

### 2.5. 失敗系の検証 (Effect-TS)

- `Effect.flip` を使用してエラーチャネルを成功チャネルに入れ替え、
  `Effect.runPromise` でエラーオブジェクトを直接受け取って検証してください。
  `Exit.match` 等の使用は冗長です。
- **スキーマ検証における異常系テスト**: `Effect.Schema` で無効な値によるデコードエラーを検証する際、
  `Schema.decodeSync` を使用して不正なデータをキャスト（`as any` 等）して渡すのではなく、
  `Schema.decodeUnknownSync` を使用してキャストなしで安全にエラーを検証してください。

## 3. 品質基準

- **カバレッジ 100% は最低条件です。**
- 「壊れたら必ず失敗する」意味のあるテストを記述してください。
- 複雑な `Match` パターンやエラー分岐は重点的に検証してください。
- 境界値（`NaN`, `Infinity`, 空文字、`null`/`undefined` 等）を必ず網羅してください。
