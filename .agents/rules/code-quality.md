# Code Quality

> [!NOTE] このファイルの役割
> プロジェクトにおける実装レベルの品質基準と、TypeScript / Effect-TS の活用規約を記載しています。

## 核心となる哲学：制約による自由

本プロジェクトでは、「正しく書くこと」を当然の前提とし、それを「個人の注意力」に依存せず「誤って書けない仕組み」によって盤石にすることを重視します。
自由すぎるコードは混沌を招きます。適切な制約こそが、堅牢でメンテナンス性の高いシステムへの近道です。

## 0. ルールの絶対性（鉄の掟）

この世界のルールは絶対であり、何よりも優先されます。

- **警告の禁止**: ESLint, TypeScript の型チェック等における「警告（Warning）」は、将来のエラーの火種であり、技術的負債です。
  **すべての警告はエラーと同様に扱い、一つ残らず解消しなければなりません。**
  警告を無視したままタスクを完了とすることは、プロフェッショナルとして許されません。
- **制約の遵守**: アーキテクチャの制約や設計指針は、絶対に破ることができない「世界の法則」です。
- **独断の禁止**: 法を曲げる、または一時的に無視することは、エージェント一人の判断で決めていいことではありません。
- **承認の義務**: この世界の法則を変えたければ、必ずユーザー（君）の承認を得ること。

## 1. 型システムによる制約

- **Make Illegal States Unrepresentable**: 存在し得ない状態を型で表現できないように設計します。
- **Opaque Types / Branded Types**: 単なる `string` や `number` ではなく、意味を持たせた型を定義し、誤った代入をコンパイルレベルで防ぎます。
- **Exhaustive Matching**: `Match` モジュールを活用し、条件分岐の漏れをコンパイルエラーとして検出します。

## 2. 厳格な値の定義

- **Total Functions**: 入力に対して必ず結果（または明示的なエラー）を返す関数を記述します。例外を throw してはいけません。
- **No `null`, Use `undefined`**: 存在しない可能性を示す値としては、原則として `undefined` を使用します。
  - より厳密な制御が必要な場合は `Effect.Option` を使用してください。
- **Primitive Obsession の回避**: 意味のあるデータの塊は、常に型定義を行い、生のプリミティブ型で持ち回さないようにします。

## 3. 宣言的かつ不変な実装

- **Immutability by Default**: すべての値は不変として扱います。オブジェクトや配列を直接操作（Mutation）することは禁止です。
  - `ReadonlyArray`, `Readonly` 型を積極的に活用してください。
- **Pure Functions**: 副作用（I/O, 状態変更）を持つ処理と、純粋なデータ変換ロジックを分離します。
- **Functional Pipelines**: `pipe` や `Effect.gen` を使用し、命令的な制御フローを排除します。

## 4. 関数型エッセンスの活用

- **Data Transformation**: 命令的な `for` ループを避け、`map`, `flatMap`, `filter`, `reduce` などの高階関数で表現します。
- **Effect-TS Integration**: 副作用の可能性はすべて `Effect` に包み込み、型システムの上で安全に管理します。
- **Validation**: 外部からの入力は、Adapter 層の境界で `Effect.Schema` を用いて即座に検証します。

## 5. エラーの明示化

- **Typed Errors**: エラーは文字列ではなく、タグ付けされたオブジェクトとして定義し、種類を型で判別可能にします。
- **Explicit Failure**: 失敗の可能性を戻り値の型（`Effect<A, E, R>` の `E`）に含め、呼び出し側に適切な対処を強制します。
- **Declarative Side-Effects**: エラー時の副作用（ログ、通知等）は `catchAll` ではなく、`Effect.tapError` を用いて宣言的に記述します。

## 6. 型システムの遵守

- **No `as any`**: テストコードであっても `as any` の使用は禁止です。型チェックを意図的に回避する場合は、`// @ts-expect-error <理由>` を使用してください。
